🏦 Financial Services Public Sector Contract ✅ Conditional Authorization Fixed-Price + T&M Hybrid GCC Jurisdiction

1.1 Client Profile & Project Background

The client is a GCC central bank responsible for monetary policy, banking supervision, and the national payments infrastructure of a Gulf Cooperation Council member state with a GDP exceeding USD 420 billion. The bank operates a core banking platform originally implemented in 2004 using a major international vendor's legacy system, connected to 48 licensed commercial banks and financial institutions through a proprietary messaging infrastructure incompatible with modern ISO 20022 payment messaging standards.

The bank's Board of Governors approved a strategic digital transformation program in 2021 mandating migration to ISO 20022-compliant messaging for all domestic interbank settlement, RTGS (Real-Time Gross Settlement), and correspondent banking operations — a prerequisite for SWIFT CBPR+ compliance by November 2025. The program also required modernization of the bank's internal IT infrastructure to support cloud-integrated monitoring, enhanced cybersecurity posture, and a future-ready API gateway for Open Banking policy implementation.

PTGT was selected through a restricted competitive tender process following pre-qualification against the bank's Information Technology Tender Requirements (ITTR-2021-044), competing against 7 international firms. PTGT's bid was awarded on the basis of its ISO 20022 implementation track record, FinCore™ middleware platform, GCC regulatory expertise, and Arabic-language delivery capability.

1.6 Security Framework

Given the systemic importance of the central bank's payment infrastructure, PTGT applied its highest security classification (Critical Financial Infrastructure — CFI Level 3) to this engagement. The security framework was aligned with the SWIFT Customer Security Programme (CSP) v2022 mandatory controls and the GCC central bank's internal IT Security Policy.

Key Security Controls Implemented:

  • Network Segmentation: RTGS core systems isolated on a physically separated network segment (air-gap) from internet-connected systems. PTGT engineers accessed development environments via a dedicated jump server with CyberArk PAM session recording and no copy-paste capability.
  • Cryptographic Standards: All message signing and verification used RSA-4096 with SHA-512; all transport encryption TLS 1.3 (TLS 1.0/1.1 explicitly disabled); symmetric encryption AES-256-GCM; key material stored exclusively in Thales Luna HSM with M-of-N split knowledge access (3-of-5 scheme).
  • SWIFT CSP Compliance: PTGT implemented all 23 mandatory SWIFT CSP controls (v2022) and 8 advisory controls, including secure zone architecture, multi-factor authentication for all SWIFT operators, and independent attestation audit completed by Deloitte UAE in October 2023.
  • Penetration Testing: Full network and application penetration test conducted by PTGT's internal red team (OSCP-certified) in August 2023 (pre-production), with critical and high findings remediated before go-live. Independent third-party retest conducted by CrowdStrike Services in September 2023.
  • SOC Integration: PTGT SOC (Casablanca) integrated into the bank's security monitoring for the duration of the hypercare period (6 months post-go-live), providing 24/7 Tier-2 security event monitoring on the new infrastructure.

1.8 Delivery Methodology

The project was delivered using PTGT's Regulated System Integration (RSI) Methodology, a hybrid framework combining PRINCE2 Agile governance with SAFe (Scaled Agile Framework) Essentials for the technical delivery teams. The RSI Methodology is specifically designed for complex, high-risk system integrations in regulated financial environments where both structured governance and delivery agility are required.

Delivery Structure:

  • Program Governance: Monthly Steering Committee (PTGT CTO + client CIO + central bank IT Director + independent advisor from GCC central bank consortium); weekly Project Board (PM level); daily scrums at team level.
  • Change Control: Formal Change Control Board (CCB) with mandatory impact assessment for all changes affecting RTGS production scope. 14 change requests processed during the project; 11 approved, 2 deferred, 1 rejected.
  • Testing Regime: Unit testing (JUnit 5, minimum 85% code coverage), integration testing (Postman/Newman automated test suites — 1,247 test cases), SIT (System Integration Testing — 6 weeks), UAT (User Acceptance Testing — 8 weeks with client operations team), parallel running (12 weeks running legacy and new system simultaneously), go-live.
  • Knowledge Transfer: Mandatory 4-week knowledge transfer program for client IT team (14 engineers) covering FinCore™ operations, Kafka cluster management, HSM administration, and incident response procedures. PTGT Academy online course access granted for 3 years.

2.6 Security Framework

Security was designed under the NEN 7510:2017 standard (equivalent to ISO 27001 with healthcare-specific controls) augmented by IHE IT Infrastructure security profiles. The platform received a full security audit by the Dutch National Cyber Security Centre (NCSC-NL) prior to launch — a standard requirement for national health IT infrastructure operated on behalf of central government.

  • Access Control: Role-based access control (RBAC) with profession-based roles (GP, specialist, pharmacist, nurse, patient, administrator). Healthcare professional identity verified via UZI-card (Dutch healthcare professional identity card) or SMART-on-FHIR institutional authorization.
  • Audit Logging: Every API access event is logged to an immutable NEN 7513-compliant audit log. Patients can view their complete access log through the patient portal, showing which healthcare provider accessed which data at what time — a statutory right under Dutch healthcare data regulations (Wgbo).
  • Incident Response: PTGT SOC integrated; DPIA-mandated 72-hour breach notification procedure documented; tested in tabletop exercise with client CISO and Dutch DPA representative in April 2023.

1. Data Residency Requirements Are Sector and Jurisdiction-Specific

All five projects required specific data residency arrangements tailored to the jurisdiction. 2. The Scope Limitation Mechanism Works as Intended

Project 4 (Smart Grid Analytics) provides a documented example of PTGT's Eligibility Decision Framework operating correctly under real contract conditions.
